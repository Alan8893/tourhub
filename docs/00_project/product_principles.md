# Назначение документа

Данный документ определяет фундаментальные принципы разработки TourHub.

Любое техническое решение, новая функция, изменение архитектуры или пользовательского интерфейса должно соответствовать данным принципам.

Если предлагаемое решение противоречит хотя бы одному принципу, оно должно быть пересмотрено.

---

# Главная цель проекта

TourHub — модульная ERP-платформа для управления туристическим клубом.

Главная задача системы — максимально автоматизировать работу клуба и сократить количество ручной работы.

TourHub должен хранить не только данные, но и знания клуба.

---

# Основные принципы

## 1. Экономия времени пользователя

Каждая новая функция должна отвечать на вопрос:

"Экономит ли она время пользователя?"

Если ответ отрицательный — функция не должна попасть в продукт.

---

## 2. Workflow First

Проектирование начинается не с базы данных.

Не с API.

Не с интерфейса.

Проектирование начинается с пользовательского сценария (User Journey).

Сначала определяется последовательность действий пользователя.

После этого появляются:

- доменные сущности;
- API;
- база данных;
- пользовательский интерфейс.

Никогда наоборот.

---

## 3. Domain First

Все изменения начинаются с предметной области.

Порядок разработки всегда одинаковый:

User Journey

↓

Domain Model

↓

ER Diagram

↓

Database

↓

Backend

↓

Frontend

---

## 4. TourHub хранит знания

TourHub хранит не только документы.

TourHub должен накапливать опыт клуба.

Например:

- комментарии инструкторов;
- рекомендации по маршрутам;
- лучшие рецепты;
- особенности приготовления;
- рекомендации по закупке;
- историю прохождения маршрутов.

---

## 5. Один источник истины

Каждая сущность существует только в одном месте.

Не допускается хранение одинаковой информации в нескольких местах.

Дублирование данных запрещено.

---

## 6. Если пользователь вводит одинаковую информацию дважды — это ошибка проектирования

Примеры:

Любимые блюда инструктора должны храниться в профиле пользователя.

Настройки экспорта должны храниться в профиле пользователя.

Название клуба должно храниться в настройках клуба.

Повторный ввод одинаковых данных недопустим.

---

## 7. Максимальная автоматизация

Если система может выполнить действие автоматически, пользователь не должен делать его вручную.

Например:

- расчет количества продуктов;
- расчет упаковок;
- расчет оборудования;
- проверка разнообразия меню;
- проверка повторяющихся блюд;
- определение сезона;
- формирование закупки;
- расчет общего веса.

---

## 8. Rule Driven

TourHub является системой, основанной на правилах.

Любые вычисления выполняются через Calculation Rules.

Запрещается писать специальную бизнес-логику для отдельных продуктов.

Недопустимо:

if product == "Свекла"

Допустимо:

CalculationRule

↓

Calculation Engine

↓

Result

---

## 9. Никакой бизнес-логики в базе данных

База данных отвечает только за хранение данных.

Все вычисления выполняются исключительно бизнес-логикой приложения.

---

## 10. Calculation Engine — единый движок расчётов

Любые расчеты выполняются через Calculation Engine.

Он используется для:

- продуктов;
- упаковок;
- оборудования;
- веса;
- газа;
- расходников;
- будущих модулей.

Calculation Engine не знает конкретных продуктов.

Он знает только правила расчета.

---

## 11. Object Card UX

Каждая сущность системы отображается в виде карточки.

Карточка содержит:

- основную информацию;
- историю изменений;
- комментарии;
- файлы;
- связанные объекты.

Пользователь работает с объектами, а не с таблицами.

---

## 12. История важнее текущего состояния

Каждый объект должен иметь историю изменений.

Пользователь должен понимать:

кто

что

когда

почему

изменил.

Удаление данных не приветствуется.

Предпочтительно использовать архивирование.

---

## 13. Soft Delete

Данные не удаляются физически.

Используется архивирование.

Это позволяет восстановить объект и сохранить историю.

---

## 14. TourHub — платформа

Веб-интерфейс является лишь одним из клиентов системы.

Архитектура должна позволять в будущем использовать:

- мобильное приложение;
- Telegram-бот;
- CLI;
- публичный API;
- интеграции.

---

## 15. Modular Monolith

TourHub разрабатывается как модульный монолит.

Каждый модуль имеет собственную ответственность.

Модули взаимодействуют только через публичные интерфейсы.

---

## 16. Минимальная связанность модулей

Ни один модуль не должен напрямую обращаться к внутренней реализации другого модуля.

Допускается только взаимодействие через публичный API модуля.

---

## 17. Feature First

Структура проекта строится вокруг бизнес-функций.

Не допускается структура вида:

controllers/

services/

helpers/

utils/

Структура строится вокруг возможностей системы.

Например:

meal_planner/

recipes/

warehouse/

projects/

---

## 18. Никаких "магических" папок

В проекте запрещены папки:

helpers

utils

common

misc

temp

other

Каждый файл должен иметь очевидное место.

---

## 19. Каждая сущность должна существовать в User Journey

Если сущность не используется ни в одном пользовательском сценарии, она не должна появляться в системе.

---

## 20. Каждая функция должна иметь бизнес-ценность

Перед разработкой любой функции необходимо ответить на вопрос:

Какую проблему пользователя она решает?

Если ответа нет — функция откладывается.

---

## 21. Документация важнее реализации

Любая архитектурная идея сначала оформляется документом.

После утверждения появляется код.

Не наоборот.

---

## 22. Никакого технического долга

Временные решения допускаются только в исключительных случаях.

Любой компромисс должен быть задокументирован через ADR.

---

## 23. Код должен читаться как книга

Названия классов, методов и переменных должны максимально отражать предметную область.

Например:

project.generate_meal_plan()

лучше чем

generate()

---

## 24. TourHub должен быть расширяемым

Новые модули должны подключаться без изменения существующего кода.

Архитектура должна быть готова к развитию в течение многих лет.

---

## 25. Простота важнее "умности"

Если существует два решения одинакового качества, выбирается более простое.

Сложность должна появляться только тогда, когда она действительно необходима.

---

# Definition of Done

Любая задача считается завершенной только если:

- реализована;
- протестирована;
- документирована;
- соответствует архитектуре;
- соответствует Product Principles;
- не создаёт технический долг;
- имеет понятный пользовательский интерфейс.

---

# Девиз проекта

Каждая функция должна экономить время инструктора.
