# TourHub Development Rules

Version: 1.0

Status: Active

Last Updated: 2026-07-03

---

# 1. Назначение документа

Данный документ определяет обязательные правила разработки проекта TourHub.

Правила распространяются на:

- Backend;
- Frontend;
- Database;
- Infrastructure;
- Documentation;
- AI Assistant;
- Code Review.

Соблюдение данных правил обязательно.

Любое исключение должно быть предварительно согласовано.

---

# 2. Главные принципы

При разработке TourHub используются следующие принципы.

## Business First

Главным является бизнес-процесс.

Код существует исключительно для реализации бизнес-логики.

Никогда нельзя проектировать систему вокруг таблиц базы данных.

Сначала моделируется бизнес.

Потом архитектура.

Потом код.

---

## Product First

Первая задача — получить рабочий продукт.

Любое усложнение должно приносить практическую пользу.

Не допускается разработка функциональности "на будущее".

---

## Simplicity

Предпочтение всегда отдаётся наиболее простому решению.

Если существуют два решения одинакового качества — выбирается более простое.

---

## Readability

Код читается значительно чаще, чем пишется.

Любой новый разработчик должен понимать код без дополнительных объяснений.

---

## Maintainability

Каждое решение оценивается с точки зрения сопровождения.

Если решение сложно сопровождать — оно считается плохим.

---

## Predictability

Во всём проекте используются одинаковые подходы.

Если один модуль реализован определённым способом, остальные должны использовать такой же подход.

---

# 3. Архитектурные правила

Используется Modular Monolith.

Используется Feature First.

Используется Vertical Slice.

Используется Clean Architecture.

Используются Domain Processes.

Используются Engines.

Архитектура проекта фиксирована.

Изменение архитектуры допускается только после обсуждения.

---

# 4. Production Ready

Каждый коммит должен быть готов к использованию.

Запрещается:

TODO

FIXME

Temporary

Mock

Hack

Stub

Временный код

Закомментированный код

Если задача не может быть завершена полностью, она должна быть разбита на более мелкие задачи.

---

# 5. Git

Используются Conventional Commits.

Каждый Commit представляет собой одну законченную задачу.

Примеры.

feat:

fix:

docs:

refactor:

test:

chore:

Не допускаются коммиты вида:

fix

update

changes

test2

new

---

# 6. Документация

Документация является частью продукта.

При изменении архитектуры обновляются:

ARCHITECTURE.md

PROJECT_CONTEXT.md

При изменении предметной области обновляется:

DOMAIN.md

При изменении процесса разработки обновляется:

DEVELOPMENT_RULES.md

Документация не должна отставать от кода.

---

# 7. AI Development

AI является полноценным участником команды.

Любой AI обязан:

изучить PROJECT_CONTEXT.md;

изучить ARCHITECTURE.md;

изучить DOMAIN.md;

изучить DEVELOPMENT_RULES.md;

только после этого приступать к написанию кода.

AI запрещается:

самостоятельно менять архитектуру;

подключать новые библиотеки;

использовать другие технологии;

менять структуру проекта.

При необходимости AI обязан сначала объяснить последствия изменений.

---

# 8. Definition of Done

Любая задача считается завершённой только если:

код написан;

код протестирован;

код проходит линтер;

код проходит форматирование;

добавлены необходимые тесты;

обновлена документация (если требуется);

проект успешно запускается.

Если хотя бы одно условие не выполнено — задача считается незавершённой.

# 9. Backend Rules

Backend является единственным источником истины.

Любая бизнес-логика реализуется исключительно на стороне Backend.

Frontend не должен принимать бизнес-решений.

---

## Структура модулей

Каждый модуль имеет одинаковую структуру.

module/
│
├── api/
├── application/
├── domain/
├── infrastructure/
└── tests/

Создание новых структур без необходимости запрещено.

---

## Application Layer

Application содержит:

- бизнес-сценарии;
- координацию работы Domain;
- взаимодействие с Engines;
- DTO.

Application не содержит SQL.

Application не содержит HTTP.

---

## Domain Layer

Domain содержит исключительно предметную область.

Domain не знает:

- FastAPI;
- SQLAlchemy;
- PostgreSQL;
- HTTP;
- JWT;
- Docker.

Domain должен иметь возможность работать как обычная Python-библиотека.

---

## Infrastructure Layer

Infrastructure отвечает исключительно за работу с внешними системами.

Например:

PostgreSQL

Redis

Файловая система

SMTP

Infrastructure запрещается принимать бизнес-решения.

---

## API Layer

API отвечает исключительно за:

получение запроса;

валидацию;

авторизацию;

вызов Application;

возврат ответа.

API запрещается:

выполнять SQL;

содержать бизнес-логику;

выполнять вычисления.

---

# 10. Engine Rules

Engine является независимым вычислительным модулем.

Engine:

не знает HTTP;

не знает FastAPI;

не знает SQLAlchemy;

не знает PostgreSQL.

Engine получает DTO.

Engine возвращает DTO.

Engine должен тестироваться отдельно.

Engine не имеет побочных эффектов.

Если возможно, Engine должен быть детерминированным.

При одинаковых входных данных Engine обязан возвращать одинаковый результат.

---

# 11. Process Rules

Process представляет собой бизнес-процесс.

Process может использовать:

Modules;

Engines;

Repositories.

Process отвечает за последовательность выполнения операций.

Process не содержит вычислительной логики.

Все вычисления выполняются Engines.

---

# 12. Repository Rules

Repository отвечает только за доступ к данным.

Repository запрещается:

валидировать бизнес-правила;

принимать решения;

выполнять расчёты.

Repository должен быть максимально простым.

---

# 13. Database Rules

Используется PostgreSQL.

Изменение схемы допускается только через Alembic.

Запрещается:

ручное изменение схемы;

ручное создание таблиц;

ручное изменение индексов.

Все изменения оформляются миграциями.

---

## Первичные ключи

Во всех таблицах используются UUID.

Последовательные integer ID не используются.

---

## Удаление данных

Физическое удаление используется только в исключительных случаях.

По возможности применяется Soft Delete.

---

## Аудит

Все основные сущности должны содержать:

created_at

updated_at

created_by

updated_by

В дальнейшем возможно добавление deleted_at.

---

# 14. Frontend Rules

Frontend отвечает только за интерфейс.

Любые вычисления выполняются Backend.

Frontend не должен дублировать бизнес-логику.

---

## Компоненты

Компоненты должны быть небольшими.

Если компонент становится слишком большим — он разделяется.

---

## API

Вызовы Backend инкапсулируются.

Запрещается выполнять fetch() непосредственно внутри страниц.

Все запросы проходят через слой API.

---

## Формы

Все формы используют React Hook Form.

Валидация должна совпадать с Backend.

---

## Таблицы

Для отображения больших списков используется AG Grid.

Не допускается самостоятельная реализация сложных таблиц.

---

# 15. Testing Rules

Каждый новый функционал сопровождается тестами.

Минимальные требования.

Backend:

Unit Tests

Frontend:

Component Tests

Engine:

Отдельные Unit Tests

Process:

Integration Tests

---

## Что обязательно тестируется

Все Engines.

Все бизнес-процессы.

Все критические API.

Все расчёты.

Все алгоритмы генерации.

---

# 16. Docker Rules

Все сервисы запускаются исключительно через Docker Compose.

Локальная установка PostgreSQL запрещена.

Локальная установка Redis запрещена.

Все окружения должны быть воспроизводимыми.

---

# 17. Configuration Rules

Конфигурация хранится в .env.

Никогда не хранить секреты в Git.

Все значения по умолчанию документируются в .env.example.

---

# 18. Logging Rules

Используется структурированное логирование.

Каждое исключение должно быть записано в журнал.

Не допускается скрывать ошибки конструкциями:

try:

except:

pass

Любое исключение должно быть обработано явно.

---

# 19. Error Handling

Все ошибки должны иметь единый формат.

Пользователь получает понятное сообщение.

Разработчик получает полную информацию в логах.

Не допускается вывод traceback пользователю.

---

# 20. Security Rules

Все проверки доступа выполняются только Backend.

Frontend используется исключительно для отображения.

Пароли никогда не хранятся в открытом виде.

Используется bcrypt.

JWT используется только через централизованный Security Layer.

---

# 21. Performance Rules

Не выполнять лишние SQL-запросы.

Избегать N+1.

Использовать индексы.

Не выполнять тяжёлые вычисления во Frontend.

При необходимости использовать Background Jobs.

---

# 22. Code Review Rules

Каждый Pull Request должен отвечать на вопросы:

Что сделано?

Почему это сделано?

Какие последствия?

Какие тесты добавлены?

Изменена ли документация?

Если архитектура изменилась — обновлены ли ARCHITECTURE.md и PROJECT_CONTEXT.md?

---

# 23. Запрещается

Запрещается:

использовать Any без причины;

игнорировать исключения;

оставлять мёртвый код;

дублировать бизнес-логику;

копировать код между модулями;

изменять архитектуру без обсуждения;

использовать временные решения;

создавать большие "God Classes";

создавать циклические зависимости;

хранить секреты в репозитории.

---

# 24. Главный принцип проекта

TourHub разрабатывается как коммерческий продукт.

Каждое решение должно оцениваться с точки зрения:

простоты;

поддерживаемости;

масштабируемости;

качества кода;

удобства сопровождения.

Если решение ухудшает хотя бы один из этих пунктов, оно требует дополнительного обсуждения.


## Technical Debt Policy

Known technical debt must be documented in TASKS.md.

Current technical debt:

- PDF generator requires refactoring into table-based document rendering.
- Document templates should become reusable components.
- Export formats should share unified presentation models.
