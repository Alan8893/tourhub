# TH-0061.6 — Recipe Components and Recipe Variants

Status: DONE

## Goal

Deliver the recipe domain required for realistic hiking menu generation while keeping the editor understandable for instructors.

## Completed Scope

- separated Dish and Recipe responsibilities at the persistence level;
- implemented RecipeComponent roles: base, cooking, optional, and serving add-on;
- implemented practical quantity modes: per person, fixed group, and package per people;
- connected RecipeComponent data to shopping calculations with legacy fallback;
- added recipe list and detail API;
- added recipe creation and renaming;
- added product catalogue reading and creation;
- added component create, update, and delete operations;
- added recipe note create, update, delete, priority ordering, and frontend editing;
- added active and archived recipe library views;
- added safe archive and restore;
- added guarded physical deletion when a dish references a recipe;
- added complete frontend recipe editor and lifecycle controls;
- added backend and frontend regression coverage and CI enforcement.

## Deferred Scope

The following depends on the later multi-user phase and is not part of this completed single-club task:

- CLUB and PERSONAL ownership scopes;
- publication review;
- verified-instructor moderation;
- invitation-based ownership and permissions.

The next product task is the dish catalogue and explicit dish-to-recipe selection workflow.

## Acceptance Result

- domain model implemented and documented;
- existing MealPlan logic remains compatible;
- shopping generation consumes component quantities;
- archived recipes remain readable for historical references;
- destructive deletion is blocked for recipes used by dishes;
- CI passes backend tests, Ruff, mypy, Alembic single-head, frontend tests/build/audit, and PostgreSQL backup/restore.
