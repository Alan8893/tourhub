# TH-0061.6 — Recipe Components and Recipe Variants Design

Status:

ACTIVE

## Goal

Design the recipe domain required for realistic hiking menu generation.

The system must support instructor-friendly recipe creation without forcing users to think like professional cooks.

---

# Context

Current Dish -> Recipe -> Product model needs extension.

Real hiking preparation uses practical quantities:

Example:

Antonsky salad:

- 1 can of beans per 4 people;
- 1 can of corn per 4 people;
- 1 can of peas per 4 people;
- half garlic head per 4 people;
- 1 pack of crackers per 2 people.

---

# Objectives

Define:

- Dish entity responsibility;
- Recipe Variant model;
- Recipe ownership;
- Recipe Components;
- hiking quantity rules;
- integration with MealPlan generation;
- integration with shopping calculation.

---

# Required Rules

## Dish != Recipe

One dish may have multiple recipe variants.

Example:

Dish:

Borscht

Variants:

- Standard system recipe;
- Instructor personal recipe with beans.

---

## Recipe Scope

Initial model:

- system recipe;
- instructor personal recipe.

No club/tenant separation.

One TourHub system instance represents one tourist club.

---

## Recipe Components

Components must support roles:

- Base Component;
- Cooking Component;
- Optional Component;
- Serving Add-on.

---

## Quantity Rules

Support hiking units:

- package;
- can;
- piece;
- portion;
- weight.

Example:

amount + unit + people_count

---

# Acceptance Criteria

- Domain model documented.
- Architecture impact reviewed.
- ADR created if required.
- Existing MealPlan logic remains compatible.
- Shopping generation can consume component quantities.

---

# Dependencies

Depends on:

- Meal Composition Rules Engine.
- Modular architecture principles.

---
