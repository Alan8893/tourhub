# TH-0061.5 — Meal Composition Rules Engine

Status: PLANNED

## Goal

Implement deterministic business rules for realistic hiking menu generation on top of the current MealPlan and MealSlot model.

The task extends generation and validation logic. It must preserve the existing persisted menu structure and purchasing recalculation contract.

## Current model

```text
MealPlan
  └─ MealPlanDay[]
       └─ MealSlot[]
            └─ MealSlotDish[]
                 └─ Dish
                      └─ selected Recipe
```

`MealPlanItem` remains a legacy compatibility path and must not be used as the primary model for new generation behavior.

No new persistence entity is required unless implementation proves that the current MealSlot structure cannot represent an approved rule. Such a change requires architecture review, migration analysis, and synchronized documentation.

## Required business rules

### Meal boundaries

- generate only meals inside the selected first-meal and last-meal range;
- support one-day trips, including the same first and last meal;
- use domain order: breakfast, snack, lunch, dinner.

### Meal composition

Breakfast:

- main dish;
- snacks or additions;
- drinks.

Lunch:

- main dish;
- snacks or additions;
- drinks.

Snack:

- fruits;
- dried fruits;
- sweets;
- cookies;
- energy bars;
- sandwiches only as a lower-priority fallback.

Dinner:

- main dish;
- salad or other addition;
- drinks.

Composition roles must be represented through existing dishes and MealSlot membership until a separately approved domain model is introduced.

### Diversity

- the same main dish should not repeat within three days;
- the same dish should not appear twice on one day;
- drinks, bread, tea, coffee, and universal additions may repeat;
- manual user choices override automatic generation rules;
- when the catalogue is insufficient, generation may repeat dishes but must return a warning.

### Selection priority

Use only metadata that actually exists in persistence. Future signals such as instructor preferences, season compatibility, dietary metadata, equipment requirements, and product availability must not be simulated or silently inferred before their domains are implemented.

When those signals become available, the approved priority is:

1. explicit instructor preference;
2. season compatibility;
3. product availability;
4. menu diversity;
5. cooking simplicity;
6. weight and transportation convenience.

## Integration requirements

- generated output must populate MealSlot and MealSlotDish structures;
- existing manually selected dishes must remain unchanged during participant-count recalculation;
- generated and edited menus must continue to feed shopping and packaging calculation;
- insufficient-catalogue warnings must be exposed through the API and frontend;
- archived recipes must not be newly selected through their dishes;
- legacy MealPlanItem compatibility must remain covered until explicitly removed.

## Implementation steps

1. Define deterministic composition and diversity rules in the Engine or Domain layer.
2. Define generation input/output DTOs around MealSlot composition.
3. Implement meal-boundary and one-day-trip handling.
4. Implement same-day uniqueness and three-day main-dish diversity.
5. Add explicit insufficient-catalogue warnings and fallback behavior.
6. Persist generated results through the existing MealPlan service boundary.
7. Add realistic unit, service, API, and regression tests.
8. Update current roadmap, status, domain, and architecture documents if contracts change.

## Definition of done

- MealSlot is the primary generated composition model.
- Approved boundary, composition, and diversity rules are implemented.
- Manual choices remain authoritative.
- Insufficient catalogue data produces a visible warning rather than an unexplained failure.
- Shopping recalculation remains correct after generation and editing.
- Legacy compatibility is either retained and tested or removed through an approved migration plan.
- All quality gates pass.
