# TH-0103 — Operational Write Audit Coverage

## Status

IN PROGRESS

## Goal

Extend the existing semantic actor-aware audit coverage to the remaining approved operational write paths without introducing ORM-wide auditing or changing the accepted single-club architecture.

## Scope

- Product catalogue create and update actions.
- Successful CSV catalogue apply actions for products and recipes, using aggregate metadata only.
- Purchase-list generation and responsible-person changes.
- Purchase-checklist generation and item progress changes.
- Recipe equipment requirements and project equipment-list generation/manual overrides.
- Successful project and purchase-list document generation/download actions.
- Russian Audit UI labels and filters for the new actions and entities.
- Focused rollback, no-op, protected-payload, actor-attribution, and browser acceptance coverage.

## Transaction rules

- Domain-write events are recorded before the owning commit and roll back with the domain mutation.
- Services invoked with `commit=False` append audit events to the caller-owned transaction.
- No-op edits do not create audit events.
- Successful document-generation events are committed before the response is returned; generated bytes and filenames are never stored in audit payloads.
- CSV content, arbitrary request bodies, generated document bytes, credentials, tokens, and exception details are excluded.

## Non-goals

- ORM-wide automatic auditing.
- Audit export, retention UI, SIEM integration, undo, or event replay.
- New product domains, asynchronous document generation, or persisted document versions.
- Database schema changes or movement of the immutable `v0.1.0` tag.

## Acceptance

- Focused Backend tests prove actor attribution, semantic bounded snapshots, no-op suppression, and rollback on audit failure.
- Existing Backend, Frontend, browser, migration, backup/restore, Docker, documentation, guided-release, operator, and final-readiness gates remain green.
- Alembic remains at `h10021`.
