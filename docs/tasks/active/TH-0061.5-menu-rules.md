# TH-0061.5 — Meal Composition Rules Engine

Status: IN PROGRESS

## Goal

Implement realistic hiking menu rules on top of the persisted MealSlot model without changing the approved architecture.

## Current model

```text
MealPlan
  └─ MealPlanDay[]
       └─ MealSlot[]
            └─ MealSlotDish[]
                 └─ Dish
                      ├─ selected Recipe
                      └─ DishMealRole[]
                           ├─ is_repeatable
                           └─ allowed MealType[]
```

MealSlot is primary. MealPlanItem remains a legacy compatibility path.

## Delivered foundation

- first and last meal boundaries;
- one-day meal ranges;
- domain order `breakfast`, `snack`, `lunch`, `dinner`;
- deterministic same-day uniqueness while unused dishes remain;
- deterministic fallback when the catalogue is exhausted;
- immediate insufficient-catalogue warning generation;
- separation of pure selection policy from persistence;
- invalid selection-count pseudo three-day cooldown removed by TH-0070;
- Meal Plan Editor UX and browser acceptance completed by TH-0065.

## Approved persisted classification model

ADR-013 defines normalized role and meal-type compatibility owned by `Dish`.

Allowed roles:

- `main`;
- `addition`;
- `drink`;
- `snack`.

Allowed meal types:

- `breakfast`;
- `snack`;
- `lunch`;
- `dinner`.

A Dish may have multiple roles. Each `(dish_id, role)` assignment has its own `is_repeatable` flag and at least one persisted compatible meal type.

Compatibility matrix:

- `main`, `addition`, and `drink` may allow breakfast, lunch, and/or dinner;
- `snack` may allow only the snack slot.

Important semantics:

- soup is `main` with explicit lunch/dinner compatibility, not a separate role;
- breakfast porridge is `main` with breakfast compatibility;
- a universal addition is `addition` with explicit main-meal compatibility and repeatability enabled;
- a multi-role Dish may have different meal compatibility for each role;
- classification does not belong to Recipe or MealSlotDish;
- unclassified dishes remain available for manual selection but are excluded from role-aware automatic selection;
- no migration may infer roles or compatibility from names, ingredients, recipes, or historical placement.

- empty-list classification clearing;
- duplicate role/type, empty, invalid, incompatible, and missing-dish rejection;
- API regression tests and unchanged existing generator behavior.

## Catalogue readiness slice

Stacked PR #61 implements and verifies:

- deterministic `GET /api/v1/dishes/catalogue-readiness`;
- required coverage of one `main` candidate for breakfast, lunch, and dinner;
- required coverage of one `snack` candidate for the snack slot;
- non-blocking recommendations for `addition` and `drink` coverage;
- active, classified, and unclassified Dish counts;
- exclusion of dishes whose selected Recipe is archived;
- structured coverage data without language-specific backend messages;
- visible Russian blocking warnings and recommendations in the Dish workspace;
- React Query refresh after role mutation without page reload;
- desktop and 360 px browser screenshots with no horizontal overflow.

The readiness threshold is intentionally minimal. Three-day diversity and larger candidate pools remain separate future policies.

## Required remaining rules

- merge PR #59, then retarget/revalidate/merge PR #60 and PR #61 in dependency order;
- classify the active deployment catalogue explicitly through the editor;
- implement breakfast, snack, lunch, and dinner composition using both dimensions;
- implement calendar-day three-day main-dish diversity;
- support explicitly repeatable drinks and additions;
- preserve manual choices as authoritative;
- exclude archived-recipe dishes from automatic selection;
- keep generation warnings visible after generation and later reads;
- preserve purchasing recalculation and legacy compatibility.

## Constraints

- do not introduce a new service or deployment boundary;
- do not add classification only as transient frontend or DTO flags;
- use only metadata that exists in persistence;
- do not select by role without checking meal-type compatibility;
- do not treat optional addition/drink gaps as blocking catalogue readiness;
- do not describe selection count as calendar-day cooldown;
- do not guess catalogue roles or compatibility automatically;
- preserve MealSlot purchasing recalculation and legacy compatibility.
