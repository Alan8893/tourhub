# TH-0061.5 — Meal Composition Rules Engine

Status: IN PROGRESS

## Goal

Implement realistic hiking menu rules on top of the persisted MealSlot model without changing the approved architecture.

## Current model

```text
MealPlan
  └─ MealPlanDay[]
       └─ MealSlot[]
            └─ MealSlotDish[]
                 └─ Dish
                      ├─ selected Recipe
                      └─ DishMealRole[]
```

MealSlot is primary. MealPlanItem remains a legacy compatibility path.

## Delivered foundation

- first and last meal boundaries;
- one-day meal ranges;
- domain order `breakfast`, `snack`, `lunch`, `dinner`;
- deterministic same-day uniqueness while unused dishes remain;
- deterministic fallback when the catalogue is exhausted;
- immediate insufficient-catalogue warning generation;
- separation of pure selection policy from persistence;
- invalid selection-count pseudo three-day cooldown removed by TH-0070;
- Meal Plan Editor UX and browser acceptance completed by TH-0065.

## Approved persisted role model

ADR-013 defines normalized `dish_meal_roles` metadata owned by `Dish`.

Allowed roles:

- `main`;
- `addition`;
- `drink`;
- `snack`.

A dish may have multiple roles. Each `(dish_id, role)` assignment has its own `is_repeatable` flag.

Important semantics:

- soup is `main`, not a separate role;
- a universal addition is `addition` with repeatability enabled;
- role metadata does not belong to Recipe or MealSlotDish;
- unclassified dishes remain available for manual selection but are excluded from role-aware automatic selection;
- no migration may infer roles from names, ingredients, or historical placement.

## Required remaining rules

- implement `dish_meal_roles` persistence and Alembic migration;
- expose roles through Dish responses and an atomic role-management endpoint;
- add catalogue classification UI;
- add catalogue-readiness validation and visible warnings;
- implement breakfast, snack, lunch, and dinner role composition;
- implement calendar-day three-day main-dish diversity;
- support explicitly repeatable drinks and additions;
- preserve manual choices as authoritative;
- exclude archived-recipe dishes from automatic selection;
- keep warnings visible after generation and later reads;
- preserve purchasing recalculation and legacy compatibility.

## Constraints

- do not introduce a new service or deployment boundary;
- do not add meal roles only as transient frontend or DTO flags;
- use only metadata that exists in persistence;
- do not describe selection count as calendar-day cooldown;
- do not guess catalogue roles automatically;
- preserve MealSlot purchasing recalculation and legacy compatibility.

## Implementation sequence

1. Add `DishMealRoleORM`, constrained role values, relationships, and Alembic migration.
2. Add repository/service/schema/API contracts and transactional replacement tests.
3. Add dish-editor role management and API integration coverage.
4. Classify the active catalogue explicitly.
5. Add readiness validation without changing current generation behavior prematurely.
6. Implement role-aware deterministic selection.
7. Implement calendar-day diversity context.
8. Preserve manual selections during regeneration.
9. Persist or reconstruct warnings.
10. Add unit, service, API, frontend, recalculation, and migration tests.
11. Synchronize current documentation.

## Definition of done

- MealSlot remains the primary composition model;
- persisted roles match ADR-013;
- approved role composition is implemented in Backend;
- three-day diversity is based on trip days, not selection count;
- manual choices remain authoritative;
- insufficient catalogue produces visible warnings;
- shopping recalculation remains correct;
- all quality gates pass.
