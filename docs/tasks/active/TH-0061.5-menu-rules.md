# TH-0061.5 — Meal Composition Rules Engine

Status: IN PROGRESS

## Goal

Implement realistic hiking menu rules on top of the persisted MealSlot model without changing the approved architecture.

## Current model

```text
MealPlan
  └─ MealPlanDay[]
       └─ MealSlot[]
            └─ MealSlotDish[]
                 └─ Dish
                      ├─ selected Recipe
                      └─ DishMealRole[]
                           ├─ is_repeatable
                           └─ allowed MealType[]
```

MealSlot is primary. MealPlanItem remains a legacy compatibility path.

## Delivered foundation

- first and last meal boundaries;
- one-day meal ranges;
- domain order `breakfast`, `snack`, `lunch`, `dinner`;
- deterministic same-day uniqueness while unused dishes remain;
- deterministic fallback when a compatible catalogue pool is exhausted;
- immediate insufficient-catalogue warning generation;
- separation of pure selection policy from persistence;
- invalid selection-count pseudo three-day cooldown removed by TH-0070;
- Meal Plan Editor UX and browser acceptance completed by TH-0065.

## Approved persisted classification model

ADR-013 defines normalized role and meal-type compatibility owned by `Dish`.

Allowed roles:

- `main`;
- `addition`;
- `drink`;
- `snack`.

Allowed meal types:

- `breakfast`;
- `snack`;
- `lunch`;
- `dinner`.

A Dish may have multiple roles. Each `(dish_id, role)` assignment has its own `is_repeatable` flag and at least one persisted compatible meal type.

Compatibility matrix:

- `main`, `addition`, and `drink` may allow breakfast, lunch, and/or dinner;
- `snack` may allow only the snack slot.

Important semantics:

- soup is `main` with explicit lunch/dinner compatibility, not a separate role;
- breakfast porridge is `main` with breakfast compatibility;
- a universal addition is `addition` with explicit main-meal compatibility and repeatability enabled;
- a multi-role Dish may have different meal compatibility for each role;
- classification does not belong to Recipe or MealSlotDish;
- unclassified dishes remain available for manual selection but are excluded from role-aware automatic selection;
- no migration may infer roles or compatibility from names, ingredients, recipes, or historical placement.

## Backend persistence slice

PR #59 implements and verifies:

- `MealRole` and `MealType` domain values;
- `DishMealRoleORM` with composite `(dish_id, role)` primary key;
- `DishMealRoleMealTypeORM` with composite `(dish_id, role, meal_type)` primary key;
- constrained role/meal-type strings, cascading composite foreign key, and per-role repeatability;
- Alembic revision `h10001` with no heuristic backfill;
- `allowed_meal_types` in Dish list and detail responses;
- transactional full classification replacement through `PUT /api/v1/dishes/{dish_id}/meal-roles`;
- empty-list classification clearing;
- duplicate role/type, empty, invalid, incompatible, and missing-dish rejection;
- API regression tests.

Quality run #197 passes Ruff, strict mypy, backend tests, Alembic single-head validation, existing frontend/browser gates, and PostgreSQL backup/restore.

## Catalogue readiness slice

PR #61 implements and verifies:

- deterministic `GET /api/v1/dishes/catalogue-readiness`;
- required coverage of one `main` candidate for breakfast, lunch, and dinner;
- required coverage of one `snack` candidate for the snack slot;
- non-blocking recommendations for `addition` and `drink` coverage;
- active, classified, and unclassified Dish counts;
- exclusion of dishes whose selected Recipe is archived;
- structured coverage data without language-specific backend messages;
- visible Russian blocking warnings and recommendations in the Dish workspace;
- React Query refresh after role mutation without page reload;
- desktop and 360 px browser screenshots with no horizontal overflow.

The readiness threshold is intentionally minimal. Three-day diversity and larger candidate pools remain separate policies.

## Required role-aware generation slice

PR #63 implements and verifies:

- pure `MealCompositionPolicy.required_role()` mapping;
- `main` as the required generated role for breakfast, lunch, and dinner;
- `snack` as the required generated role for the snack slot;
- candidate filtering by both persisted role and current meal type;
- breakfast-only porridge exclusion from lunch/dinner;
- lunch/dinner-only borscht exclusion from breakfast;
- unclassified Dish exclusion from automatic generation;
- exclusion of Dishes whose selected Recipe is archived;
- persisted empty MealSlots with explicit missing-pool warnings instead of incompatible fallback;
- per-role `is_repeatable` support during same-day selection;
- deterministic fallback plus the existing insufficient-catalogue warning after non-repeatable pool exhaustion;
- eager loading of Recipe, Dish roles, and allowed meal types for generation;
- MealSlotDish relation IDs preserved separately from Dish IDs;
- legacy MealPlanItem compatibility and existing purchasing behavior preserved.

Quality run #240 passes 178 backend tests, Ruff, strict mypy, Alembic single-head validation, all frontend/browser gates, and PostgreSQL backup/restore.

## Required remaining rules

- merge PR #63;
- classify the active deployment catalogue explicitly through the editor;
- add optional `addition` and `drink` components to generated main-meal slots;
- support explicitly repeatable drinks and additions;
- implement calendar-day three-day main-dish diversity;
- preserve manual choices as authoritative during regeneration;
- keep generation warnings visible after generation and later reads;
- preserve purchasing recalculation and legacy compatibility for multi-component generated slots.

## Constraints

- do not introduce a new service or deployment boundary;
- do not add classification only as transient frontend or DTO flags;
- use only metadata that exists in persistence;
- do not select by role without checking meal-type compatibility;
- do not use an unclassified or archived-recipe Dish as a hidden automatic fallback;
- do not treat optional addition/drink gaps as blocking catalogue readiness;
- do not describe selection count as calendar-day cooldown;
- do not guess catalogue roles or compatibility automatically;
- preserve MealSlot purchasing recalculation and legacy compatibility.

## Implementation sequence

1. Merge PR #63.

3. Add optional additions and drinks to generated main-meal slots.
4. Implement calendar-day diversity context.
5. Preserve manual selections during regeneration.
6. Persist or reconstruct generation warnings.
7. Add recalculation and end-to-end coverage for generated multi-component slots.
8. Synchronize current documentation after each merged slice.

## Definition of done

- MealSlot remains the primary composition model;
- persisted roles and meal compatibility match ADR-013;
- borscht cannot become a breakfast candidate unless explicitly classified for breakfast;
- breakfast-only porridge cannot become a lunch or dinner candidate;
- required role composition is implemented in Backend;
- optional additions and drinks follow persisted compatibility and repeatability;
- three-day diversity is based on trip days, not selection count;
- manual choices remain authoritative;
- insufficient catalogue produces visible warnings;
- shopping recalculation remains correct;
- all quality gates pass.
