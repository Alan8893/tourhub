# TH-0061.5 — Meal Composition Rules Engine

Status: IN PROGRESS

## Goal

Implement realistic hiking menu rules on top of the persisted MealSlot model without changing the approved architecture.

## Current model

```text
MealPlan
  └─ MealPlanDay[]
       └─ MealSlot[]
            └─ MealSlotDish[]
                 └─ Dish
                      └─ selected Recipe
```

MealSlot is primary. MealPlanItem remains a legacy compatibility path.

## Delivered foundation

- first and last meal boundaries;
- one-day meal ranges;
- domain order `breakfast`, `snack`, `lunch`, `dinner`;
- deterministic same-day uniqueness while unused dishes remain;
- deterministic fallback when the catalogue is exhausted;
- immediate insufficient-catalogue warning generation;
- separation of pure selection policy from persistence.

TH-0070 removes an invalid selection-based cooldown that did not represent three calendar days and had no persisted role input.

## Required remaining rules

- persisted meal-role metadata approved through domain and migration review;
- breakfast, snack, lunch, and dinner role composition;
- calendar-day three-day main-dish diversity;
- repeatable drinks and universal additions;
- manual choices override automatic generation;
- archived-recipe dishes are excluded;
- future preference modes affect priority without disabling diversity;
- warnings remain visible after generation and later reads.

## Constraints

- do not introduce a new service or deployment boundary;
- do not add `MealDishRole` only as a transient frontend or DTO flag;
- use only metadata that exists in persistence;
- do not describe selection count as calendar-day cooldown;
- preserve MealSlot purchasing recalculation and legacy compatibility.

## Implementation sequence

1. Merge and verify TH-0070.
2. Approve minimal persisted role metadata.
3. Add migration and API contracts.
4. Implement role-aware deterministic selection.
5. Implement calendar-day diversity context.
6. Preserve manual selections.
7. Persist or reconstruct warnings.
8. Add unit, service, API, frontend, and recalculation tests.
9. Synchronize current documentation.

## Definition of done

- MealSlot remains the primary composition model;
- approved role composition is implemented in Backend;
- three-day diversity is based on trip days, not selection count;
- manual choices remain authoritative;
- insufficient catalogue produces visible warnings;
- shopping recalculation remains correct;
- all quality gates pass.
