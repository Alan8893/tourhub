# TH-0061.5 — Meal Composition Rules Engine

Status: PLANNED

## Goal

Implement realistic hiking menu rules on top of the existing persistent MealSlot model.

## Current model

```text
MealPlan
  └─ MealPlanDay
       └─ MealSlot[]
            └─ Dish[]
                 └─ selected Recipe
```

No new meal persistence entity is required.

## Required rules

- meal-role composition for breakfast, lunch, snack, and dinner;
- three-day main-dish diversity;
- same-day dish uniqueness;
- repeatable drinks and universal additions;
- manual choices override automatic generation;
- future preference modes affect priority without disabling diversity;
- insufficient catalogue allows repetition only with a visible warning.

## Implementation steps

1. Define dish metadata required by composition rules.
2. Implement deterministic MealSlot selection.
3. Add diversity and same-day checks.
4. Add insufficient-catalogue warnings.
5. Preserve manual selections.
6. Add one-day and multi-day tests.
7. Verify purchasing recalculation remains consistent.

## Definition of done

- rules live in the domain or engine layer;
- existing MealSlot persistence is reused;
- warnings are exposed through API and frontend;
- tests and documentation are updated;
- existing quality gates remain green.
