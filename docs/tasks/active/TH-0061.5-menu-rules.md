# TH-0061.5 — Meal Composition Rules Engine

Status: IN PROGRESS

Last updated: 2026-07-16

## Goal

Implement realistic hiking menu rules on top of the persisted MealSlot model without changing the approved architecture.

## Current model

```text
MealPlan
  └─ MealPlanDay[]
       └─ MealSlot[]
            ├─ is_manually_edited
            └─ MealSlotDish[]
                 └─ Dish
                      ├─ selected Recipe
                      └─ DishMealRole[]
                           ├─ is_repeatable
                           └─ allowed MealType[]
```

MealSlot is primary. MealPlanItem remains a legacy compatibility path.

## Delivered foundation

- first and last meal boundaries;
- one-day meal ranges;
- domain order `breakfast`, `snack`, `lunch`, `dinner`;
- deterministic same-day uniqueness for non-repeatable assignments;
- separation of pure selection policy from persistence;
- invalid selection-count pseudo three-day cooldown removed by TH-0070;
- Meal Plan Editor UX and browser acceptance completed by TH-0065;
- responsive mobile navigation completed by PR #62.

## Persisted classification — delivered

ADR-013 defines normalized role and meal-type compatibility owned by `Dish`.

Allowed roles:

- `main`;
- `addition`;
- `drink`;
- `snack`.

Allowed meal types:

- `breakfast`;
- `snack`;
- `lunch`;
- `dinner`.

A Dish may have multiple roles. Each `(dish_id, role)` assignment has its own `is_repeatable` flag and at least one persisted compatible meal type.

Important semantics:

- soup is `main` with explicit lunch/dinner compatibility;
- breakfast porridge is `main` with breakfast compatibility;
- a universal addition is `addition` with explicit main-meal compatibility and repeatability enabled;
- a multi-role Dish may have different compatibility and repeatability for each role;
- classification belongs to Dish, not Recipe or MealSlotDish;
- unclassified dishes remain available for manual selection but are excluded from automatic generation;
- no migration or generator may infer roles from names, ingredients, recipes, history, or manual placement.

## Open stacked implementation slices

### PR #66 — calendar-day three-day diversity for `main`

- stacked on documentation PR #65;
- pure context records trip day and last selected day per non-repeatable `main` Dish;
- day-one selection is blocked on days two and three and eligible on day four;
- selection count is not used as a calendar proxy;
- only `main` receives the cross-day rule;
- explicitly repeatable `main` bypasses it;
- same-day uniqueness and compatibility filtering remain unchanged;
- exhausted required pools produce deterministic empty slots and warnings;
- pure engine, service, persistence, and public API regressions are included.

Exact-head Quality #264 passed.

### PR #67 — preserve manual MealSlots during regeneration

- stacked on PR #66;
- Alembic revision `h10002` adds non-null `MealSlot.is_manually_edited` with a safe false backfill;
- successful add, replace, and remove operations mark the complete slot as manually edited;
- removing the final dish therefore records an authoritative empty slot;
- project regeneration reuses the existing MealPlan and rebuilds its scheduled days instead of creating a duplicate plan;
- every marked slot still present in the new schedule is preserved exactly;
- unmarked slots are regenerated through the existing role-aware engine;
- preserved manual slots bypass automatic composition warnings and diversity rules;
- this bypass is intentional because MealSlotDish does not persist the role in which a manual dish is used, and the generator must not guess it;
- unclassified manual dishes remain preserved while still excluded from automatic candidates;
- MealPlanItem compatibility rows are rebuilt from the final preserved/generated composition;
- pure generator, service/persistence, manual mutation, migration, and public API regressions cover non-empty and empty manual slots, one-plan reuse, subsequent GET persistence, and normal generation of unmarked slots.

Out of scope for PR #67:

- persistence or deterministic reconstruction of generation warnings for later GET responses;
- frontend locking or role-selection UX for manual slots;
- inferring a manual dish role;
- new candidate-count thresholds or preference modes.

## Constraints

- do not introduce a new service or deployment boundary;
- use only metadata that exists in persistence;
- do not select by role without checking meal-type compatibility;
- do not treat optional addition/drink gaps as blocking readiness;
- do not describe selection count as calendar-day cooldown;
- do not guess catalogue roles or compatibility automatically;
- preserve manual MealSlot assignments and transactional purchasing recalculation;
- keep MealSlot/MealSlotDish primary while legacy MealPlanItem exists.
