# TH-0061.5 — Meal Composition Rules Engine

Status: IMPLEMENTATION COMPLETE — OPERATIONAL MAINTENANCE

Last updated: 2026-07-16

## Goal

Implement realistic hiking menu rules on top of the persisted MealSlot model without changing the approved architecture.

## Current model

```text
MealPlan
  ├─ warnings[]
  └─ MealPlanDay[]
       └─ MealSlot[]
            ├─ is_manually_edited
            └─ MealSlotDish[]
                 └─ Dish
                      ├─ selected Recipe
                      └─ DishMealRole[]
                           ├─ is_repeatable
                           └─ allowed MealType[]
```

MealSlot is primary. MealPlanItem remains a legacy compatibility path.

## Persisted classification semantics

ADR-013 defines normalized role and meal-type compatibility owned by `Dish`.

Allowed roles:

- `main`;
- `addition`;
- `drink`;
- `snack`.

Allowed meal types:

- `breakfast`;
- `snack`;
- `lunch`;
- `dinner`.

Important semantics:

- a Dish may have multiple role assignments;
- repeatability belongs to `(dish_id, role)`;
- every role assignment has explicit compatible meal types;
- classification belongs to Dish, not Recipe or MealSlotDish;
- unclassified dishes remain available manually but are excluded automatically;
- archived-recipe dishes are excluded from automatic generation;
- no migration or generator infers roles from names, ingredients, recipes, history, or manual placement;
- manual slots are authoritative because MealSlotDish does not persist the role used by a manual dish.
