# TH-0061.5 — Meal Composition Rules Engine

Status: IN PROGRESS

Last updated: 2026-07-16

## Goal

Implement realistic hiking menu rules on top of the persisted MealSlot model without changing the approved architecture.

## Current model

```text
MealPlan
  └─ MealPlanDay[]
       └─ MealSlot[]
            └─ MealSlotDish[]
                 └─ Dish
                      ├─ selected Recipe
                      └─ DishMealRole[]
                           ├─ is_repeatable
                           └─ allowed MealType[]
```

MealSlot is primary. MealPlanItem remains a legacy compatibility path.

## Delivered foundation

- first and last meal boundaries;
- one-day meal ranges;
- domain order `breakfast`, `snack`, `lunch`, `dinner`;
- deterministic same-day uniqueness for non-repeatable assignments;
- separation of pure selection policy from persistence;
- invalid selection-count pseudo three-day cooldown removed by TH-0070;
- Meal Plan Editor UX and browser acceptance completed by TH-0065;
- responsive mobile navigation completed by PR #62.

## Persisted classification — delivered

ADR-013 defines normalized role and meal-type compatibility owned by `Dish`.

Allowed roles:

- `main`;
- `addition`;
- `drink`;
- `snack`.

Allowed meal types:

- `breakfast`;
- `snack`;
- `lunch`;
- `dinner`.

A Dish may have multiple roles. Each `(dish_id, role)` assignment has its own `is_repeatable` flag and at least one persisted compatible meal type.

Compatibility matrix:

- `main`, `addition`, and `drink` may allow breakfast, lunch, and/or dinner;
- `snack` may allow only the snack slot.

Important semantics:

- soup is `main` with explicit lunch/dinner compatibility;
- breakfast porridge is `main` with breakfast compatibility;
- a universal addition is `addition` with explicit main-meal compatibility and repeatability enabled;
- a multi-role Dish may have different compatibility and repeatability for each role;
- classification belongs to Dish, not Recipe or MealSlotDish;
- unclassified dishes remain available for manual selection but are excluded from automatic generation;
- no migration or generator may infer roles from names, ingredients, recipes, or history.

## Current implementation slice

### PR #66 — calendar-day three-day diversity for `main`

- branch is stacked on documentation PR #65 and targets `docs/sync-role-aware-generation`;
- the pure selection context records the current trip day and the last selected day per non-repeatable `main` Dish;
- a non-repeatable `main` selected on day one is blocked on days two and three and becomes eligible again on day four;
- diversity is based on calendar-day distance, not on the number of intervening selections;
- the rule applies only to the `main` role;
- an explicitly repeatable `main` assignment bypasses the diversity restriction;
- existing same-day uniqueness remains unchanged for all non-repeatable assignments;
- role and meal-type compatibility remain mandatory;
- archived-recipe and unclassified dishes remain excluded from automatic generation;
- no incompatible fallback is introduced;
- exhausted required pools leave the automatic slot position empty and return the existing deterministic eligible-pool warning;
- pure policy/generator, service, persistence, and public API regression tests cover one-day and multi-day behavior, multiple candidates, repeatable main, day-four reuse, and pool exhaustion.

Out of scope for PR #66:

- preserving manual choices during regeneration;
- persisting or reconstructing warnings for later GET responses;
- new candidate-count thresholds or preference modes;
- frontend changes.

## Constraints

- do not introduce a new service or deployment boundary;
- use only metadata that exists in persistence;
- do not select by role without checking meal-type compatibility;
- do not treat optional addition/drink gaps as blocking readiness;
- do not describe selection count as calendar-day cooldown;
- do not guess catalogue roles or compatibility automatically;
- preserve manual MealSlot assignments and transactional purchasing recalculation;
- keep MealSlot/MealSlotDish primary while legacy MealPlanItem exists.
