Назначение документа

Данный документ определяет единый язык предметной области (Ubiquitous Language) проекта TourHub.

Все разработчики, дизайнеры, тестировщики и владельцы продукта используют термины только из этого документа.

Если термин отсутствует в документе — его нельзя использовать в коде, API или документации до обсуждения.

Главные принципы
1. TourHub — платформа управления туристическим клубом

TourHub не является генератором раскладки.

TourHub является модульной ERP-платформой управления туристическим клубом.

2. Главная сущность системы — Project

Любая деятельность клуба оформляется как Project.

Например:

Поход
ПВД
Слет
Учебный курс
Школа инструкторов
Соревнование
Подготовка мероприятия
3. TourHub хранит знания клуба

Система должна хранить не только данные.

Она должна накапливать опыт.

Например:

комментарии инструкторов;
лучшие места стоянок;
удачные рецепты;
рекомендации по закупке;
историю изменений.
Основные сущности
Club

Туристический клуб, использующий TourHub.

Хранит:

название;
логотип;
контакты;
настройки;
брендирование;
шаблоны документов.

В системе существует только один Club.

User

Пользователь системы.

Например:

инструктор;
администратор;
кладовщик;
бухгалтер (в будущем).
Role

Набор прав пользователя.

Роль не содержит логики.

Она объединяет Permissions.

Permission

Минимальная единица доступа.

Например:

создавать проекты;
редактировать блюда;
публиковать рецепты;
управлять складом.
Project

Главная сущность системы.

Любая деятельность клуба оформляется как Project.

Project объединяет различные модули.

Project может содержать:

маршрут;
план питания;
документы;
фотографии;
участников;
комментарии;
историю изменений;
закупку;
оборудование.
Expedition

Поход.

Является специализированным типом Project.

Хранит:

даты;
участников;
инструкторов;
маршрут;
питание.
Route

Маршрут.

Не является походом.

Маршрут может использоваться многими проектами.

Содержит:

описание;
GPX;
карты;
точки;
фотографии;
комментарии;
подрядчиков;
рекомендации;
историю прохождений.
Meal Plan

План питания проекта.

Содержит:

дни;
приемы пищи;
блюда;
закупку;
оборудование.
Meal

Конкретный прием пищи.

Например:

завтрак;
обед;
перекус;
ужин.
Recipe

Блюдо.

Например:

Борщ
Плов
Солянка

Recipe содержит:

описание;
способ приготовления;
ингредиенты;
оборудование;
категорию;
сезонность;
рейтинг;
время приготовления;
сложность.
Recipe Component

Использование продукта внутри рецепта.

Не является продуктом.

Содержит:

продукт;
правило расчета;
комментарий;
порядок приготовления;
возможность замены.
Product

Продукт.

Например:

гречка;
картошка;
тушенка;
томатная паста.

Product не знает, как он используется в рецептах.

Product Package

Вариант упаковки продукта.

Например:

Гречка

900 г
1500 г
5000 г

Томатная паста

200 г
500 г

Используется при расчете закупки.

Calculation Rule

Правило расчета количества.

Главная сущность движка раскладки.

Поддерживаемые типы:

Per Person
Per N People
Package Per N People
Fixed
Formula (зарезервировано)
Calculation Engine

Движок вычислений TourHub.

Не хранит данные.

Получает правило.

Возвращает результат.

Используется для расчета:

продуктов;
оборудования;
упаковок;
веса;
предупреждений.
Shopping List

Итоговая закупка проекта.

Формируется автоматически.

Shopping Item

Строка закупки.

Например:

Гречка

Нужно:

7,2 кг

Купить:

8 упаковок по 900 г
Equipment

Тип оборудования.

Например:

Кан
Горелка
Половник
Сковорода

Не является складской единицей.

Equipment Requirement

Использование оборудования рецептом.

Например:

Борщ

↓

Кан

1 шт

Конкретный экземпляр оборудования.

Например:

Кан №4

Инвентарный номер

Состояние

На складе
Document

Например:

PDF
Excel
Маршрутный лист
Инструкция
Comment

Комментарий.

Может принадлежать любой сущности.

Например:

маршруту;
блюду;
проекту;
продукту.
Attachment

Любой файл.

Например:

GPX;
PDF;
фото;
Excel.
Change Log

История изменений объекта.

Содержит:

кто;
когда;
что изменил.
Timeline

Лента событий объекта.

Отображается пользователю.

Например:

Создан проект

↓

Добавлено меню

↓

Изменен маршрут

↓

Добавлены фотографии

Timeline является представлением Change Log.

Template

Шаблон.

Может использоваться для:

проекта;
маршрута;
плана питания.
Dashboard

Главный экран системы.

Показывает пользователю актуальную информацию.

Workflow

Последовательность действий пользователя.

TourHub проектируется вокруг Workflow, а не вокруг CRUD.

Правила архитектуры
Rule Driven

Все вычисления выполняются через Calculation Rules.

Запрещается писать бизнес-логику вида:

if product.name == "Свекла":
Domain First

Сначала появляется доменная сущность.

Потом:

база данных;
API;
интерфейс.

Никогда наоборот.

One Source of Truth

Каждая сущность имеет одно место хранения.

Дублирование запрещено.

Knowledge Base

TourHub хранит знания клуба.

Не только документы.

Object Card UX

Любая сущность отображается как карточка.

Внутри карточки находятся:

информация;
история;
комментарии;
вложения.
Workflow First

Интерфейс проектируется вокруг действий пользователя.

Не вокруг таблиц.

ADR, которые уже приняты

Один экземпляр TourHub обслуживает один туристический клуб.

ADR-002

Используется Modular Monolith.

ADR-003

Все вычисления выполняет Calculation Engine.

ADR-004

TourHub является Rule Driven системой.
