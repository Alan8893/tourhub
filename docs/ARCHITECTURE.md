# 1. Introduction

## Цель документа

Данный документ описывает архитектуру проекта TourHub.

Он является главным техническим документом проекта и определяет:

- общую архитектуру системы;
- структуру репозитория;
- устройство Backend;
- устройство Frontend;
- взаимодействие компонентов;
- правила разработки новых модулей;
- требования к масштабируемости.

Документ является источником истины для всех разработчиков проекта.

---

# 2. Архитектурные принципы

TourHub разрабатывается как коммерческий продукт с расчетом на многолетнее развитие.

При проектировании приняты следующие принципы.

## 2.1 Простота важнее сложности

Любое архитектурное решение должно быть максимально простым.

Если задачу можно решить двумя способами, выбирается наиболее простой.

Мы сознательно избегаем избыточной архитектуры.

---

## 2.2 Product First

Главная цель проекта — получить работающий продукт.

Мы не проектируем функциональность "на будущее".

Любое усложнение архитектуры должно быть оправдано текущими задачами проекта.

---

## 2.3 Modular Monolith

TourHub реализуется как модульный монолит.

Каждый модуль имеет собственную область ответственности.

Модули должны быть максимально независимыми друг от друга.

При необходимости любой модуль можно будет выделить в отдельный сервис без переписывания бизнес-логики.

На текущем этапе использование микросервисной архитектуры запрещено.

Причины:

- увеличение сложности;
- необходимость сетевого взаимодействия;
- усложнение деплоя;
- отсутствие практической пользы для MVP.

---

## 2.4 Feature First

Структура проекта строится вокруг бизнес-функций, а не технических слоев.

Неправильно:

models/
services/
schemas/
repositories/

Правильно:

recipes/
products/
projects/
warehouse/

Внутри каждого модуля находятся все необходимые компоненты.

---

## 2.5 Vertical Slice

Каждая бизнес-функция является самостоятельным вертикальным срезом.

Например модуль Recipes содержит:

- API
- модели
- схемы
- сервисы
- репозитории
- права доступа

Все элементы находятся рядом.

Это уменьшает связанность проекта и значительно упрощает сопровождение.

---

## 2.6 Production Ready

Каждый создаваемый файл должен быть полностью готов к использованию.

Запрещается:

- TODO
- временный код
- временные модели
- заглушки
- "потом исправим"

Любая задача завершается полностью.

# 4. Основные компоненты системы

TourHub состоит из следующих логических частей.

## Web Client

React приложение.

Отвечает исключительно за пользовательский интерфейс.

Не содержит бизнес-логики.

---

## Backend

FastAPI приложение.

Содержит:

- API
- авторизацию
- бизнес-логику
- генераторы
- расчетные движки

Backend является единственным источником истины.

---

## Database

PostgreSQL.

Хранит все данные проекта.

Прямой доступ из Frontend запрещен.

---

## Engines

Отдельная подсистема вычислений.

Используется для:

- генерации меню;
- расчета раскладки;
- расчета упаковок;
- расчета оборудования;
- формирования PDF;
- формирования Excel.

Engines не зависят от HTTP.

Они могут использоваться:

- через API;
- через CLI;
- через Telegram Bot;
- через мобильное приложение.

---

## Documentation

Документация является частью проекта.

Любое архитектурное изменение сопровождается обновлением документации.

Документация хранится в репозитории.

# 5. Архитектурные ограничения

В проекте запрещается:

- использовать микросервисы;
- обращаться к базе данных напрямую из Engines;
- обращаться к базе данных из Frontend;
- хранить бизнес-логику во Frontend;
- дублировать бизнес-логику между Frontend и Backend;
- создавать циклические зависимости между модулями;
- использовать глобальное состояние без необходимости.

Все архитектурные изменения проходят предварительное обсуждение.

# 6. Backend Architecture

## Общая структура

Backend реализован как модульный монолит (Modular Monolith).

Основная задача такой архитектуры — разделить проект по бизнес-доменам, а не по техническим слоям.

Структура проекта:

backend/
└── app/
    ├── core/
    ├── shared/
    ├── modules/
    ├── engines/
    └── main.py

Каждая директория имеет строго определенную ответственность.

---

# 7. Core

Папка:

backend/app/core

Назначение:

Ядро приложения.

Core содержит инфраструктурный код, необходимый для запуска Backend.

Core НЕ содержит бизнес-логику.

Core НЕ знает ничего о рецептах, походах, продуктах или пользователях.

В Core находятся:

- конфигурация приложения;
- настройки окружения;
- подключение базы данных;
- жизненный цикл приложения;
- middleware;
- JWT;
- авторизация;
- dependency injection;
- регистрация роутеров;
- логирование;
- обработчики ошибок.

Структура:

core/
│
├── config.py
├── database.py
├── security.py
├── dependencies.py
├── lifespan.py
├── logging.py
├── exceptions.py
├── middleware.py
└── router.py

---

# 8. Shared

Папка:

backend/app/shared

Назначение:

Общие компоненты, используемые несколькими модулями.

Главное правило:

Shared НЕ является папкой "для всего подряд".

Перед помещением компонента в Shared необходимо ответить:

Используется ли этот код минимум двумя независимыми модулями?

Если нет — код остается внутри модуля.

Shared может содержать:

- BaseModel
- общие исключения
- Result
- Pagination
- BaseRepository
- BaseService
- типы
- enums
- утилиты

Запрещается переносить код в Shared "на всякий случай".

recipes/

# 10. Engines

Папка:

backend/app/engines

Engines — вычислительное ядро TourHub.

Это самая важная часть проекта.

Engine не является CRUD.

Engine не является Repository.

Engine не является API.

Engine получает данные,

выполняет вычисления,

возвращает результат.

Планируемые Engines:

planner/

calculation/

shopping/

packages/

equipment/

printing/

Каждый Engine независим.

Например

Planner Engine

получает:

- участников
- блюда
- настройки

возвращает:

готовое меню.

Calculation Engine

получает:

меню

возвращает:

общий вес продуктов.

Package Engine

получает:

продукты

возвращает:

необходимое количество упаковок.

Equipment Engine

получает:

меню

возвращает:

список оборудования.

Printing Engine

получает:

готовую раскладку

возвращает:

PDF

Excel

Печать.

main.py

main.py не должен содержать бизнес-логику.

main.py не должен содержать SQL.

main.py не должен содержать настройки.

создать FastAPI

# 12. Dependency Rules

Это один из важнейших разделов проекта.

Разрешены следующие зависимости.

Frontend

↓

Backend API

↓

Services

↓

Repositories

↓

Database

Engine

↓

Services

↓

Repositories

↓

Database

Engine может использовать Services.

Engine НЕ может использовать SQLAlchemy напрямую.

Repository знает только Database.

Service знает Repository.

API знает Service.

Frontend знает только API.

Запрещены следующие зависимости.

Frontend → Database

Frontend → SQLAlchemy

Repository → API

Repository → FastAPI

Engine → FastAPI

Engine → SQLAlchemy

Modules → Frontend

Database → Modules

# 14. Module Independence

Каждый модуль должен иметь минимальную связанность.

Например

Recipes

не должен импортировать Warehouse.

Warehouse

не должен импортировать Projects.

Если возникает необходимость совместной логики —

она выносится в Shared.

Если возникает необходимость совместного вычисления —

создается Engine.

Это позволяет легко масштабировать проект.

# 15. Frontend Architecture

## Общая концепция

Frontend является клиентом Backend.

Frontend отвечает исключительно за:

- отображение данных;
- ввод данных пользователем;
- навигацию;
- взаимодействие с API.

Во Frontend запрещено размещать бизнес-логику.

Все вычисления выполняются Backend.

# 16. API

Backend предоставляет REST API.

Frontend никогда не работает напрямую с базой данных.

Каждый модуль имеет собственный Router.

Например

/api/auth

/api/projects

/api/recipes

/api/products

/api/planner

/api/equipment

---

## Версионирование

На текущем этапе API не версионируется.

При появлении публичного API будет использоваться

/api/v1/

Repository отвечает только за работу с базой данных.

Repository запрещается:

- выполнять бизнес-логику;
- выполнять вычисления;
- принимать бизнес-решения.

# 19. Engines

Engine является вычислительным модулем.

Engine представляет собой обычный Python-пакет.

Engine может использоваться:

- Backend
- CLI
- Telegram Bot
- Mobile Client
- Background Job

Engine не должен зависеть от FastAPI.

---

## Planner Engine

Формирует меню.

Входные данные:

- количество участников;
- количество дней;
- любимые блюда;
- исключенные блюда;
- база рецептов.

Выходные данные:

готовый Meal Plan.

---

## Calculation Engine

Рассчитывает:

- общий вес продуктов;
- общий вес раскладки;
- стоимость;
- количество каждого продукта.

---

## Package Engine

На основании размеров упаковок определяет:

- необходимое количество упаковок;
- остаток продукта.

---

## Equipment Engine

Формирует:

общий список оборудования.

Например

2 кана

1 сковорода

2 половника

---

## Printing Engine

Формирует

PDF

Excel

Печатную версию
