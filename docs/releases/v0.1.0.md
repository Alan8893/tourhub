# TourHub v0.1.0 Release Notes

Status: RELEASE CANDIDATE

Release tag: `v0.1.0`

Alembic head: `h10021`

Product decision: **FEATURE FROZEN**

## Release scope

TourHub v0.1.0 is the first accepted local single-club release. One installation represents one tourist club. Multi-tenancy, microservices, paid runtime services, and self-service registration remain prohibited.

The release includes:

- local Docker Compose runtime with PostgreSQL 18, Redis, Backend, Frontend, health checks, backup, restore, restart persistence, and LAN access;
- typed System Settings for club identity, appearance, documents, modules, invitations, mail, users, and audit presentation;
- one-time Administrator bootstrap, invitation-only registration, server-owned sessions, user administration, roles, deactivation revocation, and guarded preparation access;
- working SMTP invitation delivery with a manual one-time-link fallback;
- CLUB/PERSONAL Recipe ownership, submission, rejection, publication, moderation, and immutable actor-aware moderation history;
- ordered Dish Recipe variants, three Project generation modes, and persisted assignment Recipe snapshots;
- Project creation, meal boundaries, menu generation and manual editing, recalculation, shopping, checklist, equipment, readiness, and reload-safe state;
- complete Russian Project PDF, workbook sheets `Поход`, `Меню`, `Раскладка`, `Закупка`, `Оборудование`, compatibility files, and coordinated ZIP;
- one centralized no-exceptions alcohol policy across Product, Recipe, Dish, lifecycle activation, API, and CSV import;
- deterministic archival of prohibited existing records through Alembic `h10021` while preserving historical relationships.

## Release verification

The release is eligible for tagging only when one exact merged `main` commit passes:

- Product Acceptance;
- Quality, including Backend, Frontend, Alembic single-head, and PostgreSQL backup/restore;
- Document Quality;
- Guided Release Acceptance;
- Operator Docs;
- Docker Release Runtime;
- Final Release Readiness, including PostgreSQL 18 `h10020 → h10021 → h10020 → h10021` verification.

The final workflow creates lightweight tag `v0.1.0` only after every required workflow completes successfully on the same commit SHA.

## Upgrade

Follow `docs/DEPLOYMENT_CHECKLIST.md`, `docs/INSTALLATION.md`, and `docs/UPDATING.md`. Create and retain a PostgreSQL backup before changing the release stack. The final database revision must be exactly `h10021`.

## Rollback

Production rollback is backup-based. Restore the pre-upgrade custom-format PostgreSQL dump and run the application version matching the restored database. An Alembic downgrade is verification evidence, not the normal operator rollback mechanism.

## Explicitly deferred non-blocking scope

The following capabilities remain outside v0.1.0 and do not block release:

- participant profiles, routes/GPX, warehouse balances, issue workflow, participant distribution, procurement prices, and external aggregators;
- richer Recipe technology, tags, season/diet metadata, preference weights, and per-meal Recipe switching;
- session administration, account recovery, user profile editing, asynchronous mail queues, bounce handling, advanced templates, and attachments;
- audit coverage beyond current semantic events, audit export, retention UI, SIEM, undo, and replay;
- Product/Dish archive-management UI;
- scheduled or emailed document generation, signatures, and encrypted configuration archives.

Any scope expansion requires a separate post-release task and an explicit Product Owner decision.
